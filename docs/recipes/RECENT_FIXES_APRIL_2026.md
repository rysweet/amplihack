This document tracks bug fixes and improvements to the Recipe Runner, default workflow, and related skills merged in April 2026, organized by the [Diátaxis](https://diataxis.fr/) framework.

## Week of 2026-04-17 — Workflow Condition Syntax, Step-03b Priority Chain, Multitask Paths, Step-15 Bash Hardening, and Skill Recipe Runner Alignment

### Fix: Rust Recipe Runner Rejects Python List Membership in Conditions (PR #4367)

**Problem**: Four `condition` fields in `default-workflow.yaml` used Python-style list membership (`x in ['a', 'b']`). The Rust recipe runner (`recipe-runner-rs`) does not support this syntax and raises an `LBracket` parser error at runtime, silently skipping or blocking the affected steps.

**Affected steps**: `step-07-write-tests`, `step-08-implement`, `step-08b-integration`, `checkpoint-after-implementation`.

**Fix**: All four conditions were rewritten as `!=` chains:

```yaml
# Before
condition: "task_type in ['feature', 'bugfix']"

# After
condition: "task_type != 'hotfix' and task_type != 'docs'"
```

**Rule**: Recipe `condition` fields support only `==`, `!=`, `and`, `or`, `not`, and parentheses. Never use Python-specific syntax.

### Fix: Step-03b Issue/PR Reference Extraction — 6-Level Priority Chain (PR #4368)

**Problem**: `step-03b-extract-issue-number` used ad-hoc regex matching with no defined priority order. It sometimes extracted PR numbers when issues were available, missed GitHub API closing-issue links, or failed silently when `step-03` returned a PR URL with no linked issue.

**Fix**: Redesigned with a deterministic 6-level priority chain:

| Priority | Source | Notes |
|----------|--------|-------|
| 1 | Direct issue URL in step-03 output | Most authoritative |
| 2 | GitHub API closing-issue link for the PR | API-verified |
| 3 | `#N` reference in task description | User-specified |
| 4 | `Closes #N` / `Fixes #N` in PR body | Convention-based |
| 5 | Issue number in branch name | Branch naming convention |
| 6 | PR number as fallback | When no issue found |

27 new tests cover each tier and edge cases (PR-only workflows, empty closing-issue responses, malformed references).

### Fix: Multitask Orchestrator Path Resolution (PR #4369)

**Problem**: `launch-parallel-round-1` located the `multitask-orchestrator` via a CWD-relative path. This broke in non-amplihack projects, worktree layouts, and `uvx`-based installs.

**Fix**: Updated to use `runtime_assets` resolution (respects `AMPLIHACK_HOME` and staged asset directories). Also added:

- JSONL event logging per launched task (enables post-hoc replay)
- Real-time heartbeat output for long-running parallel operations

**Rule**: Never hardcode paths to amplihack runtime assets. Always use `runtime_assets` or `${AMPLIHACK_HOME}`.

### Fix: Step-15 Commit/Push Bash Generation Hardening (PR #4370)

**Problem**: The bash script generated by `step-15-commit-push` had three failure modes in worktree layouts:

1. Nested heredocs terminated early when variable values contained the delimiter string
2. The pre-commit script was referenced via a repo-relative path, resolving incorrectly from a worktree
3. Repo root was assumed to be `$PWD`, which is false for worktrees (they live under `.claude/worktrees/`)

**Fixes**:

```bash
# 1. Process substitution instead of nested heredoc
git commit -F <(echo "$COMMIT_MSG")   # safe — content never misread as terminator

# 2. Absolute pre-commit path
REPO_ROOT=$(git rev-parse --show-toplevel)
PRE_COMMIT="$REPO_ROOT/.pre-commit-config.yaml"

# 3. Explicit repo root from git
REPO_ROOT=$(git rev-parse --show-toplevel)
```

**Rules**:
- Never nest heredocs when variable values may contain arbitrary text
- Always derive repo paths from `git rev-parse --show-toplevel`, not `$PWD`
- Test commit/push steps in both normal and worktree layouts

### Fix: Workflow Skills Now Use Recipe Runner (PR #4365)

**Problem**: The `default-workflow` and `investigation-workflow` SKILL.md files told Claude to read the workflow `.md` file and manually follow the steps. This is prompt-based enforcement — models can skip, reorder, or abbreviate steps.

**Fix**: Both skills rewritten (v2.0.0) to call `run_recipe_by_name()`:

```python
# default-workflow/SKILL.md — before
Read(file_path="~/.amplihack/.claude/workflow/DEFAULT_WORKFLOW.md")
# ... then follow 22 steps manually

# default-workflow/SKILL.md — after
run_recipe_by_name("default-workflow")
# recipe runner controls step progression; skipping is impossible
```

All three copy locations were synced: `.claude/skills/`, `amplifier-bundle/skills/`, `docs/claude/skills/`.

**Rule**: Skills that trigger multi-step workflows must call `run_recipe_by_name()`. Reading a markdown workflow file and asking the model to comply provides no enforcement guarantee.

## Earlier April 2026 — ADO Provider Support, Heredoc Security, Staging Fixes

### Fix: ADO Provider Support in Step-03 and Step-16 (PR #4205)

**Problem**: `gh issue create` and `gh pr create` are GitHub-only. Repositories hosted on Azure DevOps exited with error 1, blocking the entire workflow.

**Fix**: Added `detect_git_provider()` function that detects `dev.azure.com` / `visualstudio.com` remotes and branches to `az boards work-item create` and `az repos pr create` for ADO. Updated `step-03b` regex to match `_workitems/edit/NNNN` ADO URLs.

**Requirements for ADO**: `az` CLI installed, `az devops configure --defaults organization=... project=...` run once.

---

### Fix: Heredoc Injection in Step-16 (security commit, 2026-04-03)

**Problem**: Unquoted heredoc delimiters in step-16 caused bash to expand `$()` and backticks inside `{{task_description}}` and `{{design_spec}}` values.

**Fix**: Changed to quoted delimiters (`<<'EOFTASKDESC'`). Added numeric guard on `NEW_ITEM_ID` from `az boards work-item create`.

**Pattern rule**: All recipe bash steps embedding substituted content must use quoted heredoc delimiters (`<<'DELIMITER'`).

---

### Fix: Python Package Staging for Non-Amplihack Projects (PR #4265)

**Problem**: `from amplihack.recipes import run_recipe_by_name` raised `ModuleNotFoundError` on non-amplihack projects using `uvx`. `_stage_home_runtime_assets()` copied `.claude/` and `amplifier-bundle/` but never staged the Python package source.

**Fix**: Stage `src/amplihack/` → `~/.amplihack/src/amplihack/` during staging. Set `PYTHONPATH=${AMPLIHACK_HOME}/src` (absolute) instead of `PYTHONPATH=src` (CWD-relative).

---

### Fix: Staging Self-Copy Guard (PR #4325)

**Problem**: When `AMPLIHACK_HOME=~/.amplihack` (the default), `copytree_manifest()` and `_sync_home_runtime_directory()` both resolved source and destination to the same path, causing `shutil.SameFileError` that blocked all recipe runner agent steps.

**Fix**: Added `os.path.realpath` equality checks to both functions. When source and destination are the same real path, short-circuit with a no-op.
