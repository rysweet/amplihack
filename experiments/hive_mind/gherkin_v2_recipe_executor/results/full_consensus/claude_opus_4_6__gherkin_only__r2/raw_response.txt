## Summary

### `recipe_step_executor.py` — Implementation

| Feature | Mechanism |
|---|---|
| **Conditional execution** | `eval()` against context with safe `__builtins__={}`; missing keys → `False` |
| **Dependencies** | Topological iteration; skipped deps pass through, failed/timed-out deps propagate `"dependency_failed"` |
| **Retry + backoff** | Loop up to `max_retries+1` attempts; delays = `2^(attempt-1)` (1s, 2s, 4s…); injectable `sleep_func` |
| **Timeouts** | `ThreadPoolExecutor` with `cancel_event` so sleeping threads wake promptly; timed-out steps are **never** retried |
| **Output capture** | Successful output stored in `context[step_id]`; `{{key}}` templates resolved before execution, unresolved left literal |
| **Sub-recipes** | Child executor inherits parent context copy; `propagate_outputs` controls merge-back; sub-recipe failures are **not** retried |

Key design decision: `fail_then_succeed(N)` and `increment_counter()` auto-derive `max_retries` from the command when not explicitly set, so recipes that use these simulation commands without specifying retries still exercise the retry path correctly.
